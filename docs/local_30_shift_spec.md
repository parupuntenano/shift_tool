# 30人ローカル版 月間シフト作成システム仕様

## 目的

30人規模の現場向けに、Excel情報を読み込み、管理者が手直し前提で使える「8割完成」の月間シフト表を作る。

完璧な全自動作成ではなく、次の状態を完成ラインにする。

- 大きな破綻がない
- 必要人数をなるべく満たす
- 公休数をなるべく守る
- スキル不可を避ける
- 前月実績を見て、連勤や休み方の崩れを抑える
- 制約違反を見える化する
- 最後は管理者が表で手直しできる

## 対象運用

- ローカルPC上で動かす
- シフト作成・編集は管理者PCで行う
- 管理者ログインは残す
- スタッフ個別ログインは完成優先フェーズでは使わない
- スマホ提出、メール通知、クラウド運用は将来オプション扱い

## 残す機能

完成に必要な機能だけを残す。

- 管理者ログイン
- スタッフ管理
- 業務管理
- スキル管理
- Excel / CSV 取込
- 前月実績取込
- 個別制約
- シフト自動作成
- シフト編集
- 警告表示
- Excel / CSV 出力

## 削る・入口を閉じる機能

完成優先フェーズでは対象外にする。

- スタッフログイン
- スタッフのWebシフト提出
- スタッフのシフト確認
- 急休申請
- 未提出者一覧
- メール送信
- パスワード忘れ
- スマホ専用表示
- クラウド運用前提の機能

既存コードは一気に物理削除せず、まず画面導線とURL入口を閉じる。DBやモデルは、既存処理への影響を確認してから整理する。

## Excelから読み込む情報

30人版で必要な主データは以下。

- 社員番号
- 名前
- 公休数
- 備考 / 個別制約
- 業務
- スキル
- 前月実績
- その他、現場ごとの補助情報

取込後、管理者画面で内容を確認・修正できることを前提にする。

## 基本ワークフロー

1. 管理者がログインする
2. Excel / CSV からスタッフ・業務・スキル・制約を取り込む
3. 前月実績を取り込む
4. 対象月のシフトを自動作成する
5. 警告を見ながら管理者が手直しする
6. 保存すると警告を再計算する
7. Excel / CSV に出力する
8. 印刷・配布用に使う

## 自動作成の考え方

自動作成は「完成品を一発で出す」より、「管理者が直しやすい下書きを作る」ことを優先する。

優先順位の目安は以下。

1. スキル不可を避ける
2. 業務の必要人数をなるべく満たす
3. 公休数をなるべく合わせる
4. 前月実績を見て、月またぎの連勤・休み方を崩しすぎない
5. 個別制約にできるだけ沿う
6. 業務の偏りを抑える

ただし、全条件が同時に満たせない場合は警告として見える化し、管理者が判断できるようにする。

## 研修者と監督者の扱い

研修中の人は必要人数の1人として数えない。

研修者を配置できる条件は以下。

- 同じ業務に監督できる人がいる
- 監督者と研修者がマンツーマンになる
- その業務の必要人数が、研修者を除いた人数で満たされている

つまり、必要人数が2人の業務で「通常スタッフ1人 + 研修者1人」は不足扱いにする。

## 警告表示

警告は、管理者が手直しすべき場所を見つけるために使う。

最低限表示する警告は以下。

- 必要人数不足
- スキル不可の配置
- 公休数の不足・超過
- 出勤数 + 公休数 + 有給数 が月の日数と合わない
- 個別制約違反
- 前月実績を含めた月またぎ制約違反
- 研修者に監督者がいない
- 研修者が必要人数未達の業務に入っている
- 監督者1人に対して研修者が複数いる

## 管理者編集

シフト作成後は、管理者が表形式で編集できる。

- 業務を変更できる
- 公休・有給・空欄を変更できる
- 保存後に警告を再計算する
- 公休数、出勤数、有給数を確認できる
- Excel / CSV 出力に反映される

## 出力

出力対象はシフト本体を中心にする。

- Excel出力
- CSV出力
- 印刷しやすい月間シフト表
- 管理者向け集計は画面上で確認できればよい

## サンプルデータの完成条件

サンプルだけで一連の流れを確認できる状態にする。

- 30人
- 複数業務
- スキル差あり
- 個別制約あり
- 前月実績あり
- シフト生成できる
- 編集できる
- 警告を確認できる
- Excel / CSV 出力できる

## 実装整理の方針

オニオンアーキテクチャを意識し、責務を分ける。

- `domain`: シフト作成、制約判定、警告判定などの業務ルール
- `application`: 取込、生成、編集、出力のユースケース
- `infrastructure`: DB、Excel、CSV、Repository
- `presentation`: View、Form、Template、画面表示

画面都合の処理をシフト生成ロジックに混ぜず、生成ロジックを画面なしでもテストできる状態にする。
